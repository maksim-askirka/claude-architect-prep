<SubtopicSection domainKey="d3" index={0} title="CLAUDE.md & Memory System">

`CLAUDE.md` is Claude Code's persistent instruction system — markdown files automatically injected at session start, giving Claude project context without re-explanation each session.

**The CLAUDE.md hierarchy now has four levels:**

1. **Enterprise/managed** (`/Library/Application Support/ClaudeCode/CLAUDE.md` on macOS, `/etc/claude-code/CLAUDE.md` on Linux): Deployed by organizations. Cannot be excluded by individual user settings.
2. **Global** (`~/.claude/CLAUDE.md`): Applied to every session regardless of project.
3. **Project root** (`<project-root>/CLAUDE.md`): Project-specific conventions, build commands, architecture.
4. **Subdirectory** (`<subdir>/CLAUDE.md`): Scoped rules for a specific service or module.

More specific files take precedence on conflicts. All levels are additive — not mutually exclusive.

**New: `.claude/rules/` directory** — break instructions into focused topic files without bloating a single CLAUDE.md. Files can include YAML frontmatter with a `paths` field to scope loading to specific file patterns:

```yaml
---
paths:
  - "src/api/**/*.ts"
---
API handlers must validate all inputs before processing...
```

Path-scoped rules only load into context when Claude works with matching files — keeping context lean for unrelated work.

**New: `@import` syntax** — `@README` or `@~/.claude/my-prefs.md` imports another file inline. Content is loaded at session start. Useful for splitting large configs without a `@import` call overhead on every file read.

**New: `claudeMdExcludes`** — a glob pattern in `.claude/settings.local.json` that skips specific CLAUDE.md files. Useful in monorepos where adjacent teams' CLAUDE.md files are noisy or contradictory.

**Auto memory — Claude's half of the memory system.** CLAUDE.md contains *your* instructions. Auto memory is *Claude* writing notes for itself — build commands it discovered, debugging patterns, architecture decisions it learned mid-session.

- Stored at: `~/.claude/projects/<project>/memory/MEMORY.md`
- First 200 lines of `MEMORY.md` load each session automatically
- Topic files (e.g., `debugging.md`) load on demand when relevant
- Toggle with `/memory` command or `autoMemoryEnabled: false` in settings
- Requires Claude Code v2.1.59+

| | CLAUDE.md | Auto memory |
|-|-----------|-------------|
| Author | You | Claude |
| Contains | Instructions and rules you set | Learnings and patterns Claude discovered |
| Loaded | Every session (full file) | First 200 lines of MEMORY.md |
| Use for | Standards, architecture, workflows | Build commands, debugging insights |

<SubtopicSection domainKey="d3" index={1} title="Custom Commands & Skills">

Custom commands and skills are how you encode reusable workflows into Claude Code. Instead of typing out the same multi-step instructions for common tasks, you define them once and invoke them with a slash command or let Claude trigger them automatically based on context.

**Commands** are markdown files stored in `.claude/commands/<name>.md` within your project (project-scoped) or `~/.claude/commands/<name>.md` (global). When you type `/name` in a Claude Code session, the contents of that file are injected as a prompt. The file becomes the instructions Claude follows for that workflow.

For example, a `.claude/commands/review-pr.md` might contain:

```markdown
Review the current git diff for:
1. Security vulnerabilities
2. Performance issues
3. Test coverage gaps
4. Code style consistency with CLAUDE.md conventions

Output a structured report with severity levels.
```

Typing `/review-pr` instantly kicks off that workflow without you having to type it out.

**Skills** go further. A skill is a richer definition that includes a `description` field. This description tells Claude *when to proactively trigger the skill* — not just when explicitly invoked. If your deployment skill's description says "Use when the user asks to deploy, ship, release, or publish to production," Claude will suggest or invoke it automatically when it detects deployment-related requests.

The `Skill` tool is how Claude programmatically invokes skills within an agentic loop. An orchestrator agent can detect that a subtask matches a skill's description and invoke it via the `Skill` tool without explicit user direction.

**Command organization** matters at scale. Group related commands in subdirectories (`.claude/commands/test/unit.md`, `.claude/commands/test/integration.md`). The slash command path follows the directory structure. For global commands that apply across all projects — like a personal code review style or a standard commit message format — store them in `~/.claude/commands/`.

<SubtopicSection domainKey="d3" index={2} title="Plan Mode & Iterative Refinement">

**Plan mode** is Claude Code's safeguard against premature action. In standard mode, Claude reads files and writes code in interleaved steps as it works through a task. In plan mode, Claude reads, thinks, and plans — but does not write a single line of code until you approve the approach. For complex tasks, this is one of the most important features Claude Code offers.

Toggle plan mode with `Shift+Tab`. The visual indicator in the UI changes to show you're in plan mode. In this mode, Claude has access to read tools (Glob, Grep, Read, Bash for read-only commands) but will not call write tools like Write, Edit, or Bash commands that modify files.

When Claude has gathered enough context and is ready to present its approach, it uses the **`ExitPlanMode` tool**. This is not something you trigger — Claude triggers it when it believes it understands the task well enough to propose a concrete implementation plan. You then see the full plan and can:
- Approve it (Claude enters execution mode and implements the plan)
- Request changes ("Add error handling for the network failure case")
- Ask questions ("Why did you choose that approach over X?")
- Reject it entirely

The **`AskUserQuestion` tool** is Claude's mechanism for requesting clarification mid-plan. If Claude encounters a decision point — "Should this be a new component or extend the existing one?" — it can pause and ask rather than guess. This is preferable to Claude making an assumption that invalidates the entire plan.

Plan mode is most valuable for:
- **Multi-file refactors**: where a wrong first step causes cascading changes
- **New features with architectural implications**: where approach matters as much as implementation
- **Unfamiliar codebases**: where Claude needs to read more before writing
- **High-stakes changes**: deploys, database migrations, security-sensitive code

The investment in reviewing a plan before execution pays back in dramatically fewer iteration cycles.

<SubtopicSection domainKey="d3" index={3} title="CI/CD Integration & Batch Processing">

Claude Code is not just an interactive tool — it's a programmable component you can embed in CI/CD pipelines, scheduled jobs, and automated workflows. The CLI's headless mode and structured output options make this integration straightforward.

The `--headless` flag (or `CLAUDE_CODE_HEADLESS=true` environment variable) disables the interactive UI and runs Claude purely as a command-line tool. Combined with `-p` (short for `--print`), you get a single-shot invocation: `claude -p "Review this PR for security issues" --output-format json`. Claude processes the prompt, runs its tools, produces a result, and exits. No interactive session, no waiting for user input.

`--output-format json` structures the output as a JSON object with `result`, `cost`, `tokens`, and `tool_calls` fields. This makes Claude's output machine-parseable for downstream processing — feed the result to another tool, store it in a database, or post it as a PR comment.

**`--allowedTools`** is critical for CI/CD security. In a CI pipeline, you might want Claude to be able to read files and search the codebase but *not* write files or run arbitrary shell commands. `--allowedTools Read,Glob,Grep` enforces this. Even if the prompt or the codebase content tries to convince Claude to use other tools, `--allowedTools` prevents it at the runtime level.

Common CI/CD use cases:
- **Automated PR review**: trigger on pull_request events, run Claude with `--allowedTools Read,Glob,Grep` and a review prompt
- **Test generation**: after new code lands, run Claude to generate test cases for uncovered functions
- **Security scanning**: daily run with a security-focused prompt and structured JSON output to feed into alerting
- **Documentation updates**: detect code changes and regenerate affected documentation sections

The key principle for CI integration: **always use `--allowedTools` to enforce the principle of least privilege**. A CI Claude that can only read is far safer than one with full Bash access, even if you trust the workflow.
